Data Visualization — Seeing What Your Data Is Telling You

Why Visualization Matters

Humans are not good at reading thousands of rows of numbers. But we are extremely good at reading shapes, patterns, and colors. Visualization converts raw numbers into pictures — and that lets us spot patterns, anomalies, and trends that would be invisible in a spreadsheet.

Real-World Analogy

Imagine you have 1 million transactions written in a spreadsheet. Trying to spot fraud by reading row by row would take years. But if you plot transaction amounts on a chart and fraud transactions appear as outliers far above the average, you can see the problem in 5 seconds. That's the power of visualization.

What Dataiku Built in Day 2 — The 3 Charts

📊 Chart 1 — Bar Chart: Average Purchase Amount by Spending Type

What it shows: A bar for each spending category (Essential, Discretionary, Standard). The height of each bar = the average purchase amount in that category.

Why it's useful: You can immediately see — do fraudulent transactions happen more in luxury purchases? Are the average amounts very different between categories? This helps you understand the financial behavior patterns in your data.

How to read it: Taller bar = higher average purchase amount. If the "Discretionary" bar is twice as tall as "Essential", it means discretionary purchases cost, on average, twice as much.

In Dataiku: Open your prepared dataset → Charts tab → Select Bar chart → Drag spending_type to X-axis → Drag purchase_amount to Y-axis → Set Y-axis aggregation to Average.

🥧 Chart 2 — Pie Chart: Transaction Distribution by Risk

What it shows: Three slices — High, Medium, Low risk. The size of each slice represents the proportion of all transactions that fall into that risk level.

Why it's useful: At a glance you can see: "What percentage of our transactions are high risk?" If the High slice is 30%, that's a major red flag and needs immediate attention. If it's 2%, the risk is manageable.

How to read it: Bigger slice = more transactions in that category. The percentages should add up to 100%.

Tip: Pie charts work best when you have 3–5 categories. With more categories, slices become tiny and hard to compare. For many categories, use a bar chart instead.

🗺️ Chart 3 — Geo Map: Fraud vs Authorized Transactions by Region

What it shows: A geographic map with dots or heat regions showing where authorized and fraudulent transactions happened. Different colors represent fraud vs. authorized.

Why it's useful: Geography can be a strong fraud signal. If most fraud originates from one state but the cardholder lives in another state, that's suspicious. This chart makes geographic anomalies immediately visible.

What was needed first: This is why you created the Geo points and extracted card_state and merchant_state in Day 1! Without those columns, you can't plot anything on a map.

In Dataiku: Charts tab → Select Map (Scatter) → Drag merchant_location to the map field → Set color to flag_description (Authorized vs Fraud) → Each point on the map is one transaction, colored by whether it was fraudulent.
The Sampling Problem — Getting Accurate Charts

This is a critical concept that came up in Day 1 and applies directly to visualization.

Why the Default Sample Can Mislead You

By default, Dataiku shows you the first 10,000 rows of your dataset as a sample. For a quick preview, this is fine. But for charts, it can be dangerously misleading.

The problem: Imagine your data was sorted by date. The first 10,000 rows might be all January 2021 data. If you draw a pie chart showing fraud distribution, it only reflects January — not the full year. The chart would show a completely wrong picture.

Sample Type Comparison

❌ First 10,000 Rows (Default)

Picks the first 10K rows. If data is sorted, this only captures one time period or one category. Charts are biased and misleading.

✅ Random Sample

Picks rows randomly from across the entire dataset. Gives a representative view of the whole data. Charts reflect reality.

Best practice: For visualization, always switch to random sampling OR compute charts on the whole dataset (Analyze → Whole Data) for maximum accuracy.

In Dataiku: Dataset → Sample tab → Change "First N records" to "Random" → Set count to 10,000 → Apply. Now your charts will accurately represent the full dataset.
Types of Charts — When to Use What
📊

Bar Chart

Compare values across categories. Use when you want to ask: "Which category has the most / highest average?" Best for 2–15 categories.

🥧

Pie / Donut Chart

Show proportions of a whole. Use when you want to ask: "What percentage does each category represent?" Best for 3–5 categories.

📈

Line Chart

Show trends over time. Use when one axis is a date/time and you want to see how a value changes over time.

🔵

Scatter Plot

Show relationship between two numbers. Useful for spotting correlations — e.g., does higher purchase amount correlate with fraud?

🗺️

Plot data on a geographic map. Requires latitude/longitude or Geo point columns. Best for regional analysis.

📦

Show distribution of a number — minimum, maximum, median, quartiles. Useful for spotting outliers.

Dashboards — Packaging Insights for Your Team

What is a Dashboard?

A dashboard is a single page that displays multiple charts, tables, and key metrics together. Think of it like the dashboard of a car — it shows you speed, fuel, temperature all at once. You don't have to look at each instrument separately; everything important is in one view.

Real-World Analogy

A fraud analyst needs to check: total transactions today, fraud rate, which states have the most fraud, average transaction size by category — every morning. Instead of running 4 separate charts every day, a dashboard shows all of this automatically on one screen, updated and ready to review.

How Dashboards Work in Dataiku
1

Create Charts First

You build individual charts inside datasets (Charts tab). Each chart answers one specific question.

2

Publish Charts to a Dashboard

From any chart, click the Publish button → choose an existing dashboard or create a new one. The chart is now added as a tile (a panel inside the dashboard).

3

Arrange Tiles

Inside the dashboard, drag and resize tiles to arrange them visually. Put the most important charts at the top.

4

Add Dataset Tiles

You can also add raw datasets as tiles — this shows a sample of the actual data table directly in the dashboard. Note: this always shows a sample, not the full dataset.

5

Share with Business Users

Give colleagues AI Consumer (Viewer) access to your project. They can see and interact with the dashboard — apply filters, hover for details — without modifying anything. They don't need a Designer license.

Dashboard Features — What Users Can Do

Interactivity for Business Users

Filtering: Click on a chart element (e.g., a bar) to filter all other charts on the same dashboard to show only that category. For example, click "High Risk" on the pie chart → all other charts update to show only high-risk transactions.

Hovering: Hover over any chart element to see the exact numbers behind it. The chart shows a tooltip with precise values.

Zooming (line charts): Click and drag on a time-series chart to zoom into a specific time range.

Download: Charts and dashboards can be downloaded as images or PDFs for reports and presentations.

Key insight: Dashboards are the "last mile" of a data project — they're how the business actually uses the work you did. All your cleaning, joining, and feature creation becomes invisible to the business user. All they see is the dashboard, and they ask: "Can I filter by region?" That's why dashboards matter — they bridge the technical work and business decision-making.

Machine Learning — What It Is and Why It Exists

The Problem ML Solves

Before machine learning, humans had to write explicit rules for every decision. For fraud detection, that might be: "Flag a transaction if: amount > $1000 AND it's in a different state AND it's after midnight." The problem? Fraudsters quickly learn your rules and find ways around them. Rules also miss new fraud patterns you haven't thought of yet.

Real-World Analogy

Imagine teaching a child to recognize cats by giving them rules: "4 legs, whiskers, pointy ears, meows." That works for a typical cat. But what about a cat that lost a leg? Or a cat sleeping with ears flat? The rules break. A child who has seen 1000 cats develops an intuition — they just know it's a cat, even unusual ones. Machine learning does the same thing: it looks at thousands of examples and develops its own "intuition" (patterns) rather than following rigid rules.

The Formal Definition — In Plain English

Machine Learning is a method where you show a computer many examples of inputs and the correct outputs, and the computer figures out the relationship (pattern) between them automatically. Once trained, it can predict the output for new inputs it has never seen before.

In our fraud detection case: You show the model 1 million past transactions, each labelled "fraud" or "not fraud" (along with all their attributes like amount, location, risk level). The model finds patterns — combinations of features that predict fraud. Then you give it a new, unlabelled transaction and it predicts: fraud or not fraud?

The 3 Ingredients of Every ML Problem
Data (rows)
Each row is one observation — one transaction, one customer, one event. In our project: each row = one credit card transaction.
Features (X)
The input columns — everything the model can use to make its prediction. E.g., purchase_amount, risk_level, card_state, merchant_state, spending_type.
Target (Y)
The output column — what you're trying to predict. In our project: authorised_flag — is the transaction fraud (0) or authentic (1)?
Important: The target (Y) must be a real column in your training data. In our project, 2021 data has authorised_flag filled (0 or 1) — this is the training data. 2022 data has this column as NULL — this is what the model will predict. That's why we stacked them together and will split them apart later.
How a Model Actually "Learns" — Intuition

The Learning Process — Step by Step

You don't need to know the math, but understanding the concept helps enormously.

1

The model makes a guess

Given a transaction's features, the untrained model makes a random guess: fraud or not fraud.

2

Compare the guess to reality

The model compares its guess to the real label (authorised_flag). If the real label says "fraud" but the model said "not fraud", that's an error.

3

Adjust internal weights

The model slightly adjusts its internal parameters (weights) to make fewer errors. It learns: "transactions with purchase_amount > $500 and signature_provided = 0 are more likely fraud."

4

Repeat thousands of times

This process repeats for all training examples, thousands of times. Each iteration, the model gets a little bit better at finding the patterns in the data.

5

Model is trained — ready to predict

After training, the model has "memorised" the patterns. Now give it a new transaction it's never seen, and it predicts fraud or not fraud based on those learned patterns.

Types of Machine Learning — The Full Map

The Two Big Categories

🏷️ Supervised Learning

Definition: You have both input data (X = features) AND the correct output labels (Y = target). The model learns from labelled examples.

Analogy: A student studying with an answer key. They see the question AND the correct answer, so they can learn the pattern.

Our project: 2021 transactions have authorised_flag filled → this is supervised learning data. ✅

Has Labels Learns pattern Predicts new data

🔍 Unsupervised Learning

Definition: You only have input data (X). There are NO labels/answers. The model finds hidden structure or groups in the data on its own.

Analogy: A student given a stack of essays with no grades — they have to figure out which ones are "similar" by reading them and grouping them themselves.

Common use: Customer segmentation — "Group our customers into similar types" without telling the model what the groups should be.

No Labels Finds clusters Discovers groups

Both are supervised. The only difference is the type of answer they produce:

📐

Regression Examples

Predict house price. Predict employee salary. Predict tomorrow's stock price. Predict how many units will sell. Predict a student's exam score.

🎯

Binary Classification

Two categories only. Fraud / Not Fraud. Spam / Not Spam. Will customer churn / Won't churn. Will loan default / Won't default.

🎨

Multi-Class Classification

More than two categories. Predict dog breed (50+ breeds). Predict product category. Classify customer sentiment (positive/neutral/negative).

🔵

Clustering (Unsupervised)

Group similar customers together. No labels needed. K-Means is the most famous algorithm. Used for customer segmentation.

Why does the distinction matter? In Dataiku (and all ML tools), you must choose whether you're doing regression or classification when setting up your model. If you pick the wrong type, the model will be nonsensical — like asking for "the average" of categories like Fraud/Not Fraud. Always ask yourself: "Is my target a number or a category?" That immediately tells you which type to use.

Step 1 — Data Collection

Rows = Observations, Columns = Features

The first step is getting your data into Dataiku. We did this in Day 1 — importing from CSV files representing a Brazilian bank's transaction data. But in real projects, data comes from many sources: SQL databases, APIs, cloud storage, real-time streams.

Key questions to ask about your data:

• How many rows? (enough to train a model?)
• How many columns? (features available)
• What time period does it cover?
• Is the target column already labelled?
• Are there missing values?
• Is the data balanced? (equal fraud/not fraud?)
In our project: 1.3 million+ transactions across 2021–2022. 18 columns after joining. Target column (authorised_flag) has values 0 and 1 for 2021, and NULL for 2022 (to be predicted).
Step 2 — Exploratory Data Analysis (EDA)

Understanding Your Data Before Modelling — The Critical Step Many Skip

EDA means carefully exploring and visualizing your data before building any model. The reason: garbage in = garbage out. If you don't understand what's in your data, your model will silently fail and you won't know why.

What EDA involves:

A

Check for Missing Values

In Dataiku: open a dataset → Columns tab → look for columns with high % of errors or empty values. A column with 60% missing values is probably useless for training.

B

Check Class Balance

For fraud detection: are there equal numbers of fraud and non-fraud transactions? If 99% are "not fraud" and only 1% are "fraud", the model might just predict "not fraud" for everything and still get 99% accuracy — which is useless! This is called class imbalance and must be addressed.

C

Look for Correlations

Which features correlate with fraud? In Dataiku, look at statistics — does average purchase_amount differ between fraud and non-fraud? Do certain states have more fraud? These correlations become strong features.

D

Spot Outliers

Extreme values can confuse a model. A transaction of $1,000,000 might be an error or a genuinely unusual case. Visualization helps spot these. In Dataiku: Analyze a column → look at the histogram for far-off values.

Analogy

EDA is like a doctor reviewing a patient's full medical history before diagnosing. You wouldn't guess a diagnosis without looking at the test results first. EDA is looking at your data's "test results" before prescribing a model.

Step 3 — Feature Engineering (The Most Important Step)

Turning Raw Data Into Meaningful Signals for the Model

Raw data alone is often not very informative. Feature engineering means creating new columns from existing ones that are more predictive. This is an art form — it requires domain knowledge and creativity.

Step 4 — Train-Test Split (80/20) — The Most Important Concept in ML Evaluation

The Core Problem: How Do You Know If Your Model Actually Works?

Here's the fundamental challenge: a model trained on data will always perform extremely well on that same data — because it already "saw" those examples. But that tells you nothing about how it will perform on new, unseen data — which is the only thing that matters in real life.

Analogy — The Student and the Exam

Imagine a student who memorizes all 100 past exam questions and their answers. On those exact questions, they score 100%. But on a new exam with different questions, they fail — because they memorized, not understood. In ML, this is called overfitting. The model "memorized" the training data instead of learning the underlying pattern.

The solution: Before training, split your data into two parts:

🏋️ Training Set — 80%

The model is shown this data during training. It learns the patterns from these examples. The model can see the answers (labels) during training.

🧪 Test Set — 20%

This data is hidden from the model during training. After training, the model makes predictions on this data — and we compare predictions to actual labels to evaluate accuracy.

Visual Representation of 80/20 Split

TRAINING SET — 80% (1,040,000 rows)
TEST — 20%

Out of 1,300,000 total rows: 1,040,000 for training + 260,000 for testing

Critical rule: The test set must never be seen by the model during training. If even one test row is used in training, your evaluation becomes invalid — the model has already "memorized" it. In Dataiku, this is handled automatically when you set up the ML task — just select 80/20 and Dataiku manages it.

Understanding Overfitting vs Underfitting — Two Ways a Model Fails

OverfittingUnderfitting
What happenedModel memorized training data, learned noiseModel too simple, didn't learn enough

The gapLarge gap between train and test accuracyBoth scores are low

Good model: Training accuracy ≈ Test accuracy (both high). The gap is small, meaning the model genuinely learned the pattern — not just memorized the examples.
Step 5 — Training the Model (AutoML in Dataiku)

What Happens When You Click "Train" in Dataiku

In Dataiku, you don't need to write complex ML code. You use the Visual ML (AutoML) feature. Here's what happens behind the scenes:

1

You specify the target column

Tell Dataiku: "I want to predict authorised_flag." Dataiku sees it has values 0 and 1 → automatically identifies this as a binary classification problem.

2

Dataiku suggests features

AutoML looks at all your columns and recommends which ones to use as features. It automatically handles encoding (converting text categories like "High/Medium/Low" into numbers the model can understand).

3

Dataiku tries multiple algorithms

AutoML runs several different algorithms — Random Forest, Gradient Boosting, Logistic Regression, etc. — and finds the one that performs best on your data. You don't need to pick an algorithm manually.

4

Training happens automatically

Each algorithm trains on the 80% training data, adjusting its parameters to minimize prediction errors. Dataiku shows a progress bar.

5

Evaluation on the 20% test set

After training, Dataiku tests each model on the held-out 20% test data and reports accuracy, precision, recall, and AUC-ROC scores.

Step 6 — Evaluation Metrics — How to Know If Your Model Is Good

Common Metrics Explained Simply

MetricPlain English MeaningRangeGood Value
Accuracy Out of all predictions, what % were correct? 0–100% >85% (but watch for class imbalance!)
Precision Of all transactions the model flagged as fraud, what % were actually fraud? (avoids false alarms) 0–100% >75% for fraud detection
Recall Of all actual fraud transactions, what % did the model catch? (avoids missing fraud) 0–100% >80% for fraud detection
AUC-ROC Overall model quality regardless of threshold. 0.5 = random guessing. 1.0 = perfect model. 0.5–1.0 >0.85 is very good
Precision vs Recall — The Security Guard Analogy

High Precision, Low Recall: A very strict guard who only flags people he's 100% sure are threats. Very few false alarms, but many actual threats slip through.

High Recall, Low Precision: A paranoid guard who flags everyone even slightly suspicious. Catches all real threats but creates many false alarms, inconveniencing innocent people.

For fraud detection: You want high Recall (don't miss any fraud) but also good Precision (don't block legitimate customers unnecessarily).

Question 2 of 8

Your model achieves 99% accuracy on training data but only 61% on test data. What is this called and what does it mean?

  • Underfitting — the model didn't learn enough
  • Overfitting — the model memorized training data ✓
  • Good performance — training accuracy should be higher

A huge gap between training accuracy (99%) and test accuracy (61%) is the textbook definition of overfitting. The model memorized the training examples instead of learning the underlying pattern. Solution: use simpler model, more training data, or regularization techniques.
Question 3 of 8

Why do we split data into 80% training and 20% test sets instead of training on all the data?

  • Because 20% of the data is usually corrupted and must be excluded
  • To save computational time during training
  • To evaluate the model on data it has never seen — simulating real-world performance ✓
  • Because Dataiku requires exactly 80% to run ML
The whole point of the test set is to simulate the real world — where the model faces data it's never seen before. If you trained on all data, you'd have no way to honestly measure how it would perform on new transactions arriving tomorrow.
Question 4 of 8

You're building a customer segmentation model to group customers into "types" based on their behavior — but you have no predefined groups. What type of ML is this?

  • Supervised Regression
  • Supervised Classification
  • Unsupervised Learning (Clustering) ✓
  • Feature Engineering
Unsupervised learning has no target labels. You're asking the model to find structure in the data itself — grouping similar customers together without telling it what "groups" look like. K-Means clustering is the most common algorithm for this.
Question 5 of 8

In Dataiku, you plot a pie chart of fraud vs authorized transactions and it shows 95% authorized, 5% fraud. But you only used the default sample (first 10,000 rows). Why might this chart be misleading?

  • Pie charts can't show percentage distributions accurately
  • The first 10,000 rows may not be representative — if sorted by date, they could be from just one time period ✓
  • The chart is definitely accurate because 10,000 rows is a large sample
  • You need more colors on the chart
If the data is sorted by date and fraud only became common in the later months, the first 10,000 rows might have very little fraud — making the chart look misleadingly "safe". Always use random sampling for charts to get a representative view of the full dataset.
Question 6 of 8

What is the difference between a Feature and a Target in machine learning?

  • Features are rows, targets are columns
  • Features (X) are input columns used to make predictions; Target (Y) is the output column being predicted ✓
  • Features are numerical columns; targets are text columns
  • There is no difference — they are the same thing
Features (X) = inputs = everything the model uses to make its decision. Target (Y) = output = what the model predicts. In our project: Features = purchase_amount, risk, spending_type, states, etc. Target = authorised_flag (fraud or not).
Question 7 of 8

Why did we create the "Risk" column (High/Medium/Low) in Day 1's Prepare recipe? Couldn't the model just use the raw purchase_amount and signature_provided columns directly?

  • No, the model cannot handle numbers and binary columns simultaneously
  • It was required by Dataiku to run ML
  • The model could use raw columns, but the Risk column encodes domain knowledge — the combined signal of both factors — making it easier for the model to learn the fraud pattern ✓
  • Raw columns are deleted during training and only engineered features can be used
This is the essence of feature engineering. Yes, the model could use raw columns. But by creating the Risk column, we encoded a human expert's insight: "high amount + no signature = high risk of fraud." This gives the model a shortcut — instead of discovering this combination itself (which needs lots of data), we hand it to the model pre-computed. Good feature engineering accelerates learning.
Question 8 of 8

A colleague says: "Our fraud model has 98% accuracy, it's perfect!" You look closer and find that only 2% of all transactions are fraudulent. What is wrong with this reasoning?

  • Nothing — 98% accuracy is always excellent
  • A model that predicts "Not Fraud" for every single transaction would also get 98% accuracy — this is class imbalance ✓
  • The training set was too small
  • The model needs more features to achieve 99% accuracy
This is one of the most famous traps in ML evaluation. If 98% of data is "Not Fraud", a lazy model that ALWAYS predicts "Not Fraud" will be "98% accurate" — but completely useless! It never catches a single fraudster. For imbalanced datasets like fraud detection, Recall and AUC-ROC are much better metrics than accuracy.

An input column used by the model to make predictions. More meaningful features = better model.
Target (Y)
The output column the model tries to predict. Must exist (with values) in training data.
Supervised Learning
ML with labelled data — you have both inputs AND correct answers (labels) to learn from.
Unsupervised Learning
ML without labels — the model finds patterns/groups in data on its own. Used for clustering.
Regression
Predicts a continuous number (price, amount, temperature). Output can be any number.
Classification
Predicts a category (Fraud/Not Fraud, Yes/No, High/Medium/Low). Output is one of a fixed set of labels.
Train-Test Split
Dividing data into two parts: 80% for training the model, 20% for evaluating it on unseen data.
Overfitting
Model memorized training data — works great on training set, fails on new data. Large train/test gap.
Underfitting
Model too simple — performs poorly on both training and test data. Needs more complexity or features.
EDA
Exploratory Data Analysis — exploring, visualizing, and understanding data BEFORE building a model.
Feature Engineering
Creating new columns from existing ones to give the model better signals. The most impactful ML skill.
Accuracy
% of correct predictions out of all predictions. Can be misleading for imbalanced datasets.
Precision
Of all predicted frauds, what % were actually fraud? High precision = few false alarms.
Recall
Of all actual frauds, what % did the model catch? High recall = few missed frauds.
AUC-ROC
Overall model quality score (0.5=random, 1.0=perfect). Best metric for imbalanced datasets.
Class Imbalance
When one class (e.g., fraud) is much rarer than the other (not fraud). Makes accuracy misleading.
AutoML
Dataiku's automated ML — tries multiple algorithms, handles encoding, finds the best model for you.
Dashboard
A single page showing multiple charts/metrics together. Shared with business users for decision-making.
